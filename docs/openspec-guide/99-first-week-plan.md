# 99. План внедрения на первую неделю

## Сценарий первой недели

Команда начинает использовать OpenSpec в существующем корпоративном проекте.

Нужно сделать витрину продуктов. В полном виде это большая задача: список продуктов, карточка продукта, фильтры, доступность для разных клиентов, интеграции, аудит, журналирование и события.

Корпоративная платформа уже предоставляет стандартные механизмы:

- authentication;
- access control;
- audit;
- logging;
- отправку сообщений в корпоративный message broker;
- стандартную обработку ошибок;
- observability;
- общие API и UI-компоненты.

Приложение не реализует эти механизмы заново. Но внутри приложения остаются:

- бизнес-правила витрины;
- модель продукта;
- правила выбора продуктов для пользователя;
- permission codes и связь permissions с ролями;
- конфигурация audit events;
- выбор topics и типов сообщений;
- данные, которые приложение передаёт платформенным механизмам;
- обработка результата и ошибок платформы.

Это важная граница для OpenSpec:

```text
Платформа предоставляет механизм.
Приложение определяет бизнес-правило и конфигурацию использования механизма.
```

Например, платформа умеет проверять доступ, но приложение определяет, что для просмотра витрины нужен permission `product-showcase.read` и какие роли его получают.

Платформа умеет отправлять сообщения, но приложение определяет, какое business event создать, какие данные в него включить и в какой topic его отправить.

Платформа умеет вести audit log, но приложение определяет, какие действия витрины являются audit events и какие атрибуты нужно зафиксировать.

## Цель первой недели

Цель — не описать всю витрину и не закончить большой feature за пять дней.

Команда должна пройти подготовительную часть OpenSpec на одном маленьком реальном vertical slice и, если позволяет объём, начать или завершить реализацию.

Для первой недели выбран slice:

```text
Авторизованный пользователь видит read-only список продуктов,
доступных ему по бизнес-правилам и permissions.
```

В scope первого change входят:

- получение списка продуктов;
- отбор продуктов, доступных конкретному пользователю;
- permission для просмотра витрины;
- стандартная ошибка access denied;
- пустой результат, если доступных продуктов нет;
- необходимые logging и audit rules;
- platform configuration, необходимая этому slice;
- automated tests и manual QA для этого поведения.

В первый change не входят:

- карточка продукта;
- поиск и фильтры;
- оформление продукта;
- изменение продукта;
- административный интерфейс;
- полная модель всех будущих permissions;
- полный каталог audit events;
- описание всей корпоративной платформы;
- замена стандартных возможности платформы собственной реализацией.

Пример имени change:

```text
add-product-showcase-listing
```

## Роли на первую неделю

| Участник | Роль в первом change |
|---|---|
| Аналитик | Requirement Owner / Expected Behavior Owner |
| Разработчик 1 | Change Owner / Implementation Owner |
| Разработчик 2 | Platform Integration Reviewer / Architecture Reviewer |
| Тестировщик 1 | QA Owner |
| Тестировщик 2 | Regression Reviewer / Exploratory QA |

## День 1. Разобраться в процессе и определить границы системы

### Что происходит

Команда читает:

- `00-start-here.md`;
- `01-workflow.md`;
- `02-file-ownership.md`;
- `03-team-composition.md`;
- `04-agent-instructions.md`;
- процесс для своего типа проекта: `new-project.md` или `brownfield.md`.

После этого команда не начинает писать все OpenSpec-файлы сразу. Сначала она разбирает границу между приложением и корпоративной платформой.

Нужно ответить на вопросы:

```text
Что является бизнес-поведение приложения?
Что уже реализует платформа?
Что приложение должно сконфигурировать?
Какие platform contracts нельзя менять в рамках задачи?
Какие решения уже стандартизированы и не требуют нового ADR?
```

### Что делает аналитик

Аналитик описывает пользовательскую цель:

```text
Пользователь открывает витрину и видит только те продукты,
которые доступны ему по бизнес-правилам.
```

Он уточняет:

- кто является пользователем витрины;
- что означает «доступный продукт»;
- какие состояния продукта видимы;
- зависит ли доступность от роли, сегмента, региона или договора;
- что пользователь видит при отсутствии продуктов;
- какое поведение считается ошибкой.

Аналитик не определяет техническую реализацию access control, audit или messaging.

### Что делает разработчик 1

Разработчик 1 создаёт карту затронутой области:

- где хранятся продукты;
- какой компонент получает список;
- какой platform SDK используется для access control;
- где лежит конфигурация приложения;
- какие стандартные logging и audit adapters используются;
- есть ли обязательные события и topics;
- какие автоматические тесты уже есть.

### Что делает разработчик 2

Разработчик 2 проверяет корпоративные стандарты и platform contracts.

Он фиксирует:

- какие capabilities нужно переиспользовать;
- какие configuration conventions обязательны;
- где заканчивается ответственность приложения;
- какие архитектурные решения уже закреплены платформой или существующими ADR;
- где действительно может понадобиться новое решение приложения.

### Что делает тестировщик 1

Тестировщик 1 определяет основной проверяемый путь:

```text
пользователь с permission открывает витрину
→ приложение получает продукты
→ применяет бизнес-правила доступности
→ возвращает ожидаемый список
```

Он выясняет, как в тестовой среде задаются роли, permissions и platform configuration.

### Что делает тестировщик 2

Тестировщик 2 составляет первый список рисков:

- permission отсутствует;
- permission есть, но доступных продуктов нет;
- продукт выключен или не опубликован;
- конфигурация роли неполная;
- platform access service недоступен;
- данные продукта повреждены;
- пользователь относится к нескольким сегментам;
- platform configuration отличается между средами.

### Результат дня

У команды есть:

- выбранный vertical slice;
- scope и out of scope;
- схема ответственности «платформа / конфигурация приложения / бизнес-логика приложения»;
- список неизвестных;
- назначенные Change Owner, Reviewer и QA Owner.

Код в первый день писать не обязательно.

## День 2. Создать change и описать proposal

### Что происходит

Разработчик 1 создаёт:

```text
openspec/changes/add-product-showcase-listing/
├── proposal.md
├── design.md
├── tasks.md
└── specs/
```

Команда заполняет `proposal.md`.

Proposal должен объяснять проблему и scope, но не пересказывать устройство корпоративной платформы.

Пример Why:

```text
У пользователя нет единой read-only витрины продуктов,
которая учитывает бизнес-доступность и корпоративный access control.
```

Пример What Changes:

```text
Приложение предоставляет список доступных продуктов.
Просмотр требует permission product-showcase.read.
Приложение использует стандартный platform access control.
Необходимые permission mappings, audit rules и messaging configuration
хранятся в конфигурации приложения.
```

### Что делает аналитик

Аналитик является owner бизнес-части proposal:

- problem;
- user value;
- scope;
- out of scope;
- success criteria;
- affected user roles.

### Что делает разработчик 1

Разработчик 1 описывает affected areas и platform dependencies. Он не добавляет собственную реализацию возможности платформы в scope.

### Что делает разработчик 2

Разработчик 2 проверяет, что proposal не смешивает три разных вещи:

```text
business feature
platform mechanism
конфигурация приложения
```

### Что делает тестировщик 1

Тестировщик 1 проверяет, можно ли по success criteria построить acceptance test plan.

### Что делает тестировщик 2

Тестировщик 2 проверяет out of scope и добавляет риски regression.

### Результат дня

Создан OpenSpec change. `proposal.md` понятен всей команде и не обещает реализацию всей витрины за одну задачу.

## День 3. Описать specification

### Что происходит

Команда пишет delta spec для бизнес-поведение приложения.

Specification отвечает на вопрос:

```text
Что должна делать витрина?
```

Specification не должна подробно объяснять:

```text
Как устроен platform access control?
Как platform audit service сохраняет записи?
Как message broker доставляет сообщения?
```

Но наблюдаемое поведение и значимая конфигурация приложения должны быть отражены.

Пример requirements:

```md
### Requirement: Доступ к витрине продуктов

Система SHALL разрешать пользователю с необходимым permission просматривать витрину продуктов.

#### Scenario: У пользователя есть permission для просмотра витрины

- GIVEN у пользователя есть permission `product-showcase.read`
- WHEN пользователь запрашивает витрину продуктов
- THEN система возвращает продукты, доступные этому пользователю

#### Scenario: У пользователя нет permission для просмотра витрины

- GIVEN у пользователя нет permission `product-showcase.read`
- WHEN пользователь запрашивает витрину продуктов
- THEN система отклоняет запрос стандартной ошибкой платформы
```

Отдельные scenarios описывают:

- доступные продукты;
- исключение недоступных или неопубликованных продуктов;
- пустую витрину;
- отсутствие permission;
- обязательный audit event, если он требуется правилами компании;
- business event, если он является частью контракта приложения.

Не нужно добавлять audit event или message только ради демонстрации интеграции с платформой. Они включаются, если это требуется корпоративным стандартом или бизнес-процессом.

### Что делает аналитик

Аналитик пишет requirements и основной ожидаемое поведение. Он отвечает за смысл «продукт доступен пользователю».

### Что делает разработчик 1

Разработчик 1 проверяет реализуемость и связь requirements с реальными application и platform contracts.

### Что делает разработчик 2

Разработчик 2 проверяет, что specification описывает поведение, а не внутреннее устройство platform SDK.

### Что делает тестировщик 1

Тестировщик 1 проверяет тестируемость каждого scenario и готовит acceptance checks.

### Что делает тестировщик 2

Тестировщик 2 добавляет edge cases, но не расширяет scope до полной витрины.

### Результат дня

Есть reviewed draft delta spec с requirements и scenarios для первого vertical slice.

## День 4. Подготовить design и tasks

### Что происходит

Разработчики описывают технический подход в `design.md`.

Design явно разделяется на три части:

```text
1. Application business logic
2. Platform capabilities
3. Application-owned platform configuration
```

Пример содержания:

```text
Application business logic:
- получение продуктов;
- проверка статуса публикации;
- применение правил сегмента и доступности;
- формирование response model.

Platform capabilities:
- получение identity;
- проверка permission;
- audit transport;
- logging transport;
- message broker client;
- standard error mapping.

Application configuration:
- permission product-showcase.read;
- role-to-permission mappings;
- audit event codes и attributes;
- topic names и message types;
- зависящий от environment values.
```

Design также описывает:

- API impact;
- data sources;
- ошибки platform dependencies;
- security impact;
- logging и observability;
- rollout и rollback;
- automated test strategy;
- необходимость ADR.

Новое ADR не нужно, если команда просто следует существующему platform standard. ADR нужен, если приложение принимает собственное долгосрочное решение, например вводит единый `ProductAvailabilityService` как обязательную точку применения правил доступности.

После design команда заполняет `tasks.md`.

Задачи должны включать не только код:

```text
[ ] согласовать requirements и scenarios
[ ] добавить permission в конфигурация приложения
[ ] настроить role-to-permission mappings
[ ] реализовать бизнес-правила доступности
[ ] подключить стандартный platform access check
[ ] добавить audit configuration, если требуется
[ ] добавить topic configuration и event mapping, если требуется
[ ] добавить automated tests
[ ] выполнить manual QA
[ ] выполнить regression
[ ] выполнить Design Harvest
[ ] выполнить archive
```

### Что делает аналитик

Аналитик проверяет, что design не изменил бизнес-поведение и не добавил скрытые ограничения.

### Что делает разработчик 1

Разработчик 1 является owner `design.md` и `tasks.md`. Он описывает решение и разбивает реализацию на проверяемые задачи.

### Что делает разработчик 2

Разработчик 2 проводит architecture и platform integration review. Он проверяет, что команда переиспользует стандартные capabilities и не переносит бизнес-правила в platform configuration без необходимости.

### Что делает тестировщик 1

Тестировщик 1 добавляет задачи на test data, permissions, основной test plan и QA sign-off.

### Что делает тестировщик 2

Тестировщик 2 добавляет regression для соседних API, ролей, конфигураций и environment differences.

### Результат дня

Есть draft `design.md` и `tasks.md`. Команда понимает, что реализуется в приложении, что предоставляет платформа и что настраивается приложением.

## День 5. Провести review и принять решение о реализации

### Что происходит

Команда проводит review всей цепочки:

```text
proposal
→ delta spec
→ design
→ tasks
```

Review проводится до основной реализации. Для первого change разумно выделить отдельную встречу на 60–90 минут.

Команда проверяет:

- одинаково ли понимается scope;
- все ли бизнес-правила отражены в specification;
- правильно ли проведена граница с платформой;
- перечислена ли конфигурация приложения;
- определены ли permissions, audit events и topics;
- понятны ли failure scenarios;
- достаточен ли test plan;
- нет ли попытки описать или переделать всю платформу;
- нужен ли ADR;
- можно ли начинать реализацию.

### Что делает аналитик

Аналитик даёт approval ожидаемое поведение или возвращает requirements на доработку.

### Что делает разработчик 1

Разработчик 1 закрывает замечания, обновляет artifacts и подтверждает готовность tasks.

### Что делает разработчик 2

Разработчик 2 даёт architecture approval или фиксирует блокирующие вопросы по platform contracts.

### Что делает тестировщик 1

Тестировщик 1 подтверждает, что scenarios проверяемы и среду можно подготовить.

### Что делает тестировщик 2

Тестировщик 2 подтверждает regression scope и exploratory checks.

### Результат дня

Нормальный результат первой недели:

```text
OpenSpec change создан.
Proposal, delta spec, design и tasks прошли review.
Неизвестные закрыты или явно назначены владельцам.
Команда получила go для реализации.
```

Если slice действительно маленький и все platform dependencies доступны, команда может начать реализацию раньше и к пятнице открыть PR. Но PR, QA и archive не нужно объявлять завершёнными только ради выполнения недельного плана.

## Что происходит после первой недели

После go команда продолжает обычный OpenSpec cycle:

```text
implementation
→ automated tests
→ PR Review
→ manual QA
→ regression
→ Design Harvest
→ archive
```

После archive:

- master spec содержит актуальное бизнес-поведение витрины;
- archived change хранит контекст и детали реализации;
- ADR хранит новое архитектурное решение, если оно действительно появилось;
- platform documentation не дублируется в master spec;
- конфигурация приложения остаётся связанной с requirements и design.

## Definition of Done первой недели

```text
[ ] команда прочитала базовые документы OpenSpec
[ ] выбран один небольшой реальный vertical slice
[ ] определены scope и out of scope
[ ] назначены владельцы change, requirements, review и QA
[ ] проведена граница между платформой и приложением
[ ] возможности платформы перечислены
[ ] application-owned configuration перечислена
[ ] бизнес-правила приложения сформулированы
[ ] change add-product-showcase-listing создан
[ ] proposal.md подготовлен
[ ] delta spec содержит проверяемые scenarios
[ ] design.md разделяет business logic, возможности платформы и configuration
[ ] tasks.md покрывает implementation, testing, review и archive
[ ] review до реализации проведён
[ ] принято явное решение go/no-go
```

Implementation, PR, QA и archive входят в Definition of Done самого change, но не являются обязательным условием завершения первой недели внедрения OpenSpec.

## Что не считать успешным внедрением

Плохо:

```text
Команда заполнила шаблоны формально, но начала код до согласования поведения.
В specification скопирована техническая документация платформы.
Business rules спрятаны в design или конфигурация приложения.
Platform capabilities реализованы в приложении повторно.
Permissions, topics и audit events появились только в коде и конфигурации без review.
Change искусственно архивирован в пятницу без завершённого QA.
```

Хорошо:

```text
Команда выбрала маленький business slice.
Expected behavior описан отдельно от технической реализации.
Граница платформы и приложения понятна.
Application configuration рассмотрена как часть изменения.
Change прошёл review до кода.
Реализация начинается с согласованными scenarios и понятным test plan.
```

## Главное правило первой недели

Не нужно доказывать, что OpenSpec позволяет за пять дней закончить большую feature.

Нужно показать, что перед реализацией команда умеет согласовать поведение, границы платформы, конфигурация приложения, технический подход и проверки.
