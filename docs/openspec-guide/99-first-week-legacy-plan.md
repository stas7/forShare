# План первой недели: доработка существующей legacy-системы

## Сценарий

В production уже работает витрина продуктов. Она создавалась до внедрения OpenSpec, а её поведение описано частично.

Нужно доработать витрину: показывать продукты с учётом сегмента клиента.

Корпоративная платформа предоставляет стандартные механизмы:

- authentication;
- access control;
- audit;
- logging;
- message broker;
- observability;
- стандартную обработку ошибок.

Приложение использует эти механизмы, но самостоятельно определяет:

- бизнес-правила доступности продуктов;
- permission codes и role mappings;
- правила определения сегмента клиента;
- audit events и их attributes;
- topics, message types и payload;
- конфигурация приложения;
- реакцию на недоступность платформенных сервисов.

В legacy-коде уже есть несколько способов получить продукты. Часть проверок выполняется в domain service, часть — в controllers, часть — в UI. Конфигурация ролей различается между средами. Некоторые правила известны только команде поддержки.

## Главная мысль

Доработка legacy-системы начинается не с переписывания кода и не с описания всей существующей системы.

Сначала команда должна определить:

```text
Current behavior
Что система действительно делает сейчас.

Expected behavior
Что система должна делать после изменения.

Preserved behavior
Что должно остаться без изменений.

Unknowns
Что пока не подтверждено.
```

Код является источником информации о текущее поведение, но не является автоматическим доказательством правильного требования.

OpenSpec change должен зафиксировать только область, необходимую для безопасной доработки.

## Цель первой недели

Цель первой недели — подготовить один небольшой legacy-change к безопасной реализации.

Пример имени change:

```text
add-segment-based-product-availability
```

В scope входят:

- текущий способ получения списка продуктов;
- определение сегмента клиента;
- фильтрация продуктов по сегменту;
- существующий permission для просмотра витрины;
- необходимые изменения конфигурация приложения;
- поведение при неизвестном сегменте;
- audit и logging rules, относящиеся к изменению;
- regression существующей витрины.

В scope не входят:

- полная переработка каталога продуктов;
- унификация всех legacy access checks;
- новая административная панель;
- изменение корпоративной платформы;
- документирование всех ролей и permissions системы;
- миграция всех старых integrations;
- рефакторинг всех способов получения продуктов.

## Роли

| Участник | Роль в change |
|---|---|
| Аналитик | Current / Expected Behavior Owner |
| Разработчик 1 | Change Owner / Implementation Owner |
| Разработчик 2 | Legacy Architecture Reviewer / Platform Integration Reviewer |
| Тестировщик 1 | Baseline Verification Owner / QA Owner |
| Тестировщик 2 | Regression Reviewer / Exploratory QA |

## День 1. Ограничить change и собрать источники

### Что происходит

Команда определяет минимальный vertical slice и собирает источники информации о существующем поведении.

Источниками могут быть:

- production behavior;
- код и configuration;
- automated tests;
- API contracts;
- данные и ограничения базы;
- логи и audit log;
- dashboards и alerts;
- старые задачи;
- документация;
- знания поддержки и пользователей.

Для каждого важного утверждения желательно понимать, чем оно подтверждено.

### Что делает аналитик

Аналитик описывает цель доработки:

```text
Пользователь должен видеть только продукты,
доступные его клиентскому сегменту.
```

Он выясняет:

- какие сегменты существуют;
- кто определяет сегмент;
- может ли у клиента не быть сегмента;
- может ли продукт быть доступен нескольким сегментам;
- какие правила имеют приоритет;
- что должно происходить при неполных данных.

### Что делает разработчик 1

Разработчик 1 строит карту выполнения запроса:

```text
UI
→ API endpoint
→ access check
→ product service
→ database / integration
→ availability filtering
→ response
```

Он отмечает все места, где уже применяются permissions, фильтры, logging и audit.

### Что делает разработчик 2

Разработчик 2 проверяет:

- нет ли альтернативных API endpoints;
- не дублируется ли бизнес-правило;
- какие platform contracts обязательны;
- какие конфигурация приложенияs различаются между средами;
- какие существующие ADR ограничивают решение.

### Что делает тестировщик 1

Тестировщик 1 собирает основной baseline:

- пользователь с permission видит витрину;
- пользователь без permission получает отказ;
- опубликованные продукты отображаются;
- неопубликованные продукты скрываются;
- пустая витрина обрабатывается корректно.

### Что делает тестировщик 2

Тестировщик 2 ищет альтернативные пути и скрытые зависимости:

- прямые ссылки;
- старые версии API;
- разные роли;
- разные environments;
- кэш;
- повторные запросы;
- ошибки интеграций;
- продукты с неполными данными.

### Результат дня

Есть scope, out of scope, карта затронутой области, список источников и список unknowns.

## День 2. Подтвердить текущее поведение и regression baseline

### Что происходит

Команда воспроизводит существующее поведение до изменения. Наблюдения фиксируются отдельно от ожидаемое поведение.

Удобный формат:

| Случай | Current behavior | Подтверждение | Нужно сохранить |
|---|---|---|---|
| Есть permission | Витрина открывается | QA + integration test | Да |
| Нет permission | Возвращается 403 | QA + API contract | Да |
| Продукт не опубликован | Не отображается | Код + QA | Да |
| Сегмент отсутствует | Показываются все продукты | Только код | Требует решения |

Если источники противоречат друг другу, команда не выбирает удобный вариант молча. Противоречие становится отдельным вопросом к аналитику и владельцу бизнеса.

### Что делает аналитик

Аналитик классифицирует найденное поведение:

```text
подтверждённое правило
поведение, которое нужно изменить
неизвестное поведение
вероятный дефект
техническая деталь без бизнес-смысла
```

### Что делает разработчик 1

Разработчик 1 подтверждает текущее поведение по коду и данным. Для критичных веток он добавляет characterization tests, если это возможно.

Characterization test фиксирует фактическое legacy-поведение до изменения. Такой тест не утверждает, что поведение идеально; он защищает команду от случайного изменения соседних правил.

### Что делает разработчик 2

Разработчик 2 независимо проверяет спорные выводы и выявляет места, где текущее поведение является случайным следствием реализации.

### Что делает тестировщик 1

Тестировщик 1 выполняет baseline checks и сохраняет тестовые данные, роли, конфигурацию и фактические результаты.

### Что делает тестировщик 2

Тестировщик 2 проводит exploratory testing и формирует regression scope.

### Результат дня

Current behavior подтверждён в достаточном объёме. Определено, что нужно сохранить, изменить или дополнительно согласовать.

## День 3. Создать OpenSpec change и описать поведение

### Что происходит

Создаётся структура:

```text
openspec/changes/add-segment-based-product-availability/
├── proposal.md
├── design.md
├── tasks.md
└── specs/
```

В `proposal.md` описываются:

- Why;
- Current behavior;
- Expected behavior;
- What Changes;
- out of scope;
- affected areas;
- platform dependencies;
- risks;
- unknowns.

В delta spec описывается нормативное поведение после изменения.

Пример:

```md
### Requirement: Доступность продуктов по сегменту клиента

Система SHALL возвращать пользователю только продукты, доступные сегменту его клиента.

#### Scenario: Продукт доступен сегменту клиента

- GIVEN клиент пользователя относится к сегменту `business`
- AND продукт доступен сегменту `business`
- WHEN пользователь запрашивает витрину продуктов
- THEN система включает продукт в результат

#### Scenario: Продукт недоступен сегменту клиента

- GIVEN клиент пользователя относится к сегменту `business`
- AND продукт доступен только сегменту `retail`
- WHEN пользователь запрашивает витрину продуктов
- THEN система не включает продукт в результат

#### Scenario: Сегмент клиента не определён

- GIVEN для клиента пользователя не определён сегмент
- WHEN пользователь запрашивает витрину продуктов
- THEN система возвращает согласованный безопасный результат
- AND система записывает установленное диагностическое событие
```

Фраза «согласованный безопасный результат» должна быть заменена конкретным правилом до review. Это пример неизвестного, а не допустимая финальная формулировка requirement.

### Что делает аналитик

Аналитик является owner ожидаемое поведение и принимает решение для неизвестного сегмента, конфликтующих правил и приоритетов.

### Что делает разработчик 1

Разработчик 1 создаёт change и связывает requirements с существующими application и platform contracts.

### Что делает разработчик 2

Разработчик 2 проверяет, что specification не описывает внутреннее устройство платформы и не закрепляет случайные legacy-детали.

### Что делает тестировщик 1

Тестировщик 1 проверяет каждый Scenario на тестируемость и связь с baseline.

### Что делает тестировщик 2

Тестировщик 2 добавляет edge cases и regression Scenarios в границах scope.

### Результат дня

Созданы proposal и delta spec. Current behavior, ожидаемое поведение и preserved behavior разделены явно.

## День 4. Подготовить design и migration plan

### Что происходит

`design.md` описывает безопасный путь от legacy-реализации к новому поведению.

Design должен содержать:

```text
Current architecture
Где сейчас применяются правила и configuration.

Proposed solution
Где будет находиться правило сегментной доступности.

Platform capabilities
Какие стандартные механизмы переиспользуются.

Application configuration
Какие permissions, mappings, audit events и topics меняются.

Compatibility
Как сохраняется существующее поведение.

Migration
Нужна ли миграция данных или configuration.

Rollout
Feature flag, порядок включения и environments.

Rollback
Как безопасно отключить изменение.

Observability
Какие logs, metrics и alerts подтверждают корректность.
```

Пример разделения ответственности:

```text
Application business logic:
- определяет сегмент клиента;
- применяет правила доступности продукта;
- формирует список продуктов.

Platform capabilities:
- проверяет permission;
- доставляет audit event;
- пишет стандартные technical logs;
- публикует message в broker.

Application configuration:
- связывает роли с permission;
- задаёт audit event code;
- задаёт topic name;
- содержит зависящий от environment values.
```

Не нужно создавать ADR только потому, что используется стандартный platform SDK. ADR нужен, если change создаёт новое долговременное архитектурное правило приложения.

### Что делает аналитик

Аналитик проверяет, что migration и fallback не меняют ожидаемое поведение незаметно.

### Что делает разработчик 1

Разработчик 1 является owner design. Он описывает реализацию, configuration changes, data impact, rollout и rollback.

### Что делает разработчик 2

Разработчик 2 проводит legacy architecture и platform integration review. Он проверяет, что решение не создаёт ещё один дублирующий способ фильтрации.

### Что делает тестировщик 1

Тестировщик 1 определяет тестовые данные для разных сегментов и проверки feature flag.

### Что делает тестировщик 2

Тестировщик 2 проверяет риски кэша, миграции, environment configuration и обратной совместимости.

### Результат дня

Есть reviewed draft design, понятный migration plan, rollout, rollback и решение о необходимости ADR.

## День 5. Подготовить tasks и провести review до реализации

### Что происходит

Команда создаёт `tasks.md`.

Пример задач:

```text
[ ] подтвердить текущее поведение и regression baseline
[ ] согласовать правило для неизвестного сегмента
[ ] завершить requirements и scenarios
[ ] добавить модель сегментной доступности
[ ] реализовать единое применение бизнес-правило
[ ] обновить конфигурация приложения
[ ] добавить или изменить permission mappings
[ ] добавить audit configuration, если требуется
[ ] обновить topic configuration, если требуется
[ ] добавить characterization tests
[ ] добавить automated tests нового поведения
[ ] выполнить migration rehearsal, если требуется
[ ] провести PR Review
[ ] выполнить manual QA
[ ] выполнить regression
[ ] проверить metrics и logs после rollout
[ ] выполнить Design Harvest
[ ] обновить ADR, если требуется
[ ] выполнить archive
```

После этого команда проводит review цепочки:

```text
текущее поведение
→ proposal
→ delta spec
→ design
→ tasks
```

### Что делает аналитик

Аналитик утверждает ожидаемое поведение и подтверждает, что неизвестные закрыты.

### Что делает разработчик 1

Разработчик 1 закрывает замечания и подтверждает готовность implementation tasks.

### Что делает разработчик 2

Разработчик 2 даёт architecture approval или фиксирует блокирующие legacy-риски.

### Что делает тестировщик 1

Тестировщик 1 подтверждает baseline, acceptance test plan и готовность тестовых данных.

### Что делает тестировщик 2

Тестировщик 2 утверждает regression scope и exploratory checks.

### Результат дня

Нормальный результат первой недели:

```text
Legacy-область исследована в необходимом объёме.
Current behavior и preserved behavior подтверждены.
Expected behavior согласован.
OpenSpec change прошёл review.
Migration, rollout и rollback понятны.
Команда получила go для реализации.
```

Команда может начать реализацию раньше, если неизвестные закрыты. Но нельзя объявлять change завершённым только ради недельного срока.

## Реализация после первой недели

Дальше change проходит обычный цикл:

```text
implementation
→ automated tests
→ PR Review
→ manual QA
→ regression
→ controlled rollout
→ production verification
→ Design Harvest
→ archive
```

Во время реализации:

- изменение поведения требует обновления delta spec;
- изменение технического подхода требует обновления design;
- новый legacy-риск добавляется в tasks и test plan;
- platform configuration проходит такой же review, как код;
- временный workaround получает владельца и срок удаления.

## Design Harvest и archive

Перед archive команда проверяет, какие знания должны остаться после change.

В master spec попадает:

- актуальное правило сегментной доступности;
- поведение при неизвестном сегменте;
- обязательное наблюдаемое поведение access control;
- связанные business events, если они являются контрактом.

В ADR попадает:

- новое важное архитектурное решение приложения;
- выбранный подход;
- причины выбора;
- долгосрочные последствия.

В archived change остаются:

- исходное legacy-поведение;
- результаты исследования;
- migration details;
- rollout и rollback;
- рассмотренные альтернативы;
- tasks;
- delta spec.

В master spec не нужно копировать внутреннюю документацию корпоративной платформы.

## Definition of Done первой недели

```text
[ ] выбран небольшой legacy vertical slice
[ ] scope и out of scope определены
[ ] назначены владельцы
[ ] источники текущее поведение собраны
[ ] карта затронутой области подготовлена
[ ] текущее поведение подтверждён
[ ] preserved behavior определён
[ ] unknowns перечислены и имеют владельцев
[ ] regression baseline подготовлен
[ ] characterization tests добавлены, если возможно и разумно
[ ] OpenSpec change создан
[ ] proposal.md описывает current и ожидаемое поведение
[ ] delta spec содержит проверяемые Scenarios
[ ] текст после Requirement, Scenario, GIVEN, WHEN, THEN написан на русском
[ ] design.md учитывает legacy-ограничения
[ ] возможности платформы и конфигурация приложения разделены
[ ] migration, rollout и rollback описаны
[ ] tasks.md покрывает полный цикл
[ ] review до реализации проведён
[ ] принято решение go/no-go
```

Implementation, QA, rollout и archive входят в Definition of Done change, но не обязаны завершиться в первую неделю.

## Типичные ошибки

Плохо:

```text
Код объявлен единственным правильным требованием.
Команда начала с полного рефакторинга legacy-модуля.
Непонятное поведение молча перенесено в master spec.
Business rules остались разбросаны между UI, controllers и configuration.
Platform configuration изменена без review и regression.
Characterization tests заменили согласование ожидаемое поведение.
Change архивирован до production verification.
```

Хорошо:

```text
Исследована только затронутая область.
Факты отделены от предположений.
Current behavior отделён от ожидаемое поведение.
Сохраняемое поведение защищено regression baseline.
Платформа переиспользуется, а не реализуется заново.
Application configuration рассматривается как часть change.
Реализация начинается после review specification и design.
```

## Главное правило

Legacy-change должен улучшать систему в двух направлениях одновременно:

```text
изменить нужное бизнес-поведение
+
оставить актуальное проверяемое знание в OpenSpec
```

Не нужно документировать или переписывать всю legacy-систему. Нужно безопасно изменить выбранную область и сделать её понятнее, чем она была до change.
