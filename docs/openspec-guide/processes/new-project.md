# Процесс: новый проект

## Главная мысль

В новом проекте не нужно сразу пытаться описать всю будущую систему.

OpenSpec лучше внедрять постепенно:

```text
маленький первый scope
→ первый change
→ первая master spec
→ первый PR
→ первый QA
→ первый archive
```

Цель первого цикла — не сделать идеальную документацию, а научить команду проходить полный путь от идеи до archive.

## Для какой команды рассчитан процесс

Базовый состав:

```text
1 аналитик
2 разработчика
2 тестировщика
```

Роли:

| Участник | Роль в процессе |
|---|---|
| Аналитик | Requirement Owner |
| Разработчик 1 | Change Owner |
| Разработчик 2 | Architecture Reviewer / PR Reviewer |
| Тестировщик 1 | QA Owner |
| Тестировщик 2 | QA Reviewer / Exploratory QA |

## Что считается новым проектом

Новый проект — это ситуация, когда:

- кодовая база только создаётся;
- master specification ещё нет;
- архитектурные решения только формируются;
- команда ещё не договорилась о постоянных правилах;
- первые домены системы только появляются.

Примеры:

```text
новый backend-сервис
новый frontend-модуль
новая внутренняя платформа
новый микросервис
новый продуктовый MVP
```

## Главный риск нового проекта

Главный риск — попытаться заранее описать всю систему.

Плохо:

```text
Сначала полностью описываем всю систему, все модули, все API, все роли, все интеграции.
Потом когда-нибудь начинаем писать код.
```

Хорошо:

```text
Берём небольшой вертикальный slice.
Описываем только его.
Реализуем.
Проверяем.
Архивируем.
После этого берём следующий slice.
```

## Что должно остаться после первого change

После завершения первого change у команды должны остаться не только код и архивированный change.

Должны появиться три вида постоянного знания:

```text
1. Master Spec
   Что система делает.

2. ADR — Architecture Decision Record
   Почему принято важное техническое или архитектурное решение.

3. Archived Change
   Полная история конкретного изменения: proposal, design, tasks, delta spec.
```

Важно: `design.md` целиком не переносится в master spec.

Но ключевые решения из `design.md` не должны потеряться. Если решение влияет на будущую разработку, его нужно вынести в ADR.

Пример:

```text
Поведение:
Пользователь может создать заявку в статусе Draft.
→ это идёт в master spec.

Решение:
Все изменения статуса заявки проходят только через RequestStatusTransitionService.
→ это идёт в ADR.

История:
Почему в первом change выбрали именно такой endpoint, какие были ограничения, какие риски обсуждали.
→ это остаётся в archived change.
```

В master spec допускается добавлять короткую ссылку на ADR, если без неё будущей команде будет трудно понять ограничение.

Пример:

```md
### Requirement: Request Status Transition

The system SHALL change request status only through validated status transitions.

Related decision: ADR-001 Request status transitions must use transition service.
```

Master spec всё равно остаётся документом про поведение, но может ссылаться на ADR, если архитектурное решение важно для понимания правила.

## Хороший первый scope

Хороший первый scope должен быть маленьким, но полноценным.

Он должен проходить через всю цепочку:

```text
requirement
→ design
→ implementation
→ PR
→ QA
→ archive
```

Примеры хорошего первого scope:

```text
создание заявки
создание пользователя
базовая авторизация
создание справочника
первый статусный переход
первый API endpoint
```

## Плохой первый scope

Плохой первый scope слишком большой или слишком абстрактный.

Примеры:

```text
описать всю систему
реализовать весь workflow
создать всю архитектуру
сделать весь backend
сделать весь frontend
сделать все интеграции
```

## Пошаговый процесс

## Шаг 1. Определить первый рабочий slice

### Что происходит

Команда выбирает первую маленькую часть системы, которую можно реально реализовать и проверить.

### Что делает аналитик

Аналитик формулирует бизнес-смысл первого slice:

- зачем он нужен;
- кто пользователь;
- какое действие должен выполнить пользователь или система;
- какой результат должен получиться;
- что точно не входит в первый slice.

### Что делает разработчик 1

Разработчик 1 помогает понять, можно ли этот slice реализовать как самостоятельное изменение.

Он проверяет:

- не слишком ли scope большой;
- есть ли технические зависимости;
- можно ли сделать это без половины будущей системы.

### Что делает разработчик 2

Разработчик 2 смотрит на будущую архитектуру.

Он задаёт вопросы:

- не закладываем ли мы плохое архитектурное решение;
- не создаём ли временное решение, которое сразу станет постоянным;
- какие ограничения нужно зафиксировать до кода.

### Что делает тестировщик 1

Тестировщик 1 проверяет, можно ли этот slice протестировать.

Он задаёт вопросы:

- как понять, что задача сделана;
- какие сценарии нужно проверить вручную;
- какие тестовые данные понадобятся.

### Что делает тестировщик 2

Тестировщик 2 ищет неочевидные случаи:

- неправильные входные данные;
- повторные действия;
- ошибки доступа;
- граничные статусы;
- нестандартные пользовательские сценарии.

### Результат шага

Команда выбрала первый небольшой scope.

Пример:

```text
Пользователь может создать заявку в статусе Draft.
```

## Шаг 2. Создать OpenSpec structure

### Что происходит

В репозитории создаётся базовая структура OpenSpec.

```text
openspec/
├── specs/
└── changes/
```

### Что делает аналитик

Аналитик проверяет, что в будущей структуре будет понятно, где описывать бизнес-домены.

Например:

```text
approval-requests
users
roles
notifications
```

### Что делает разработчик 1

Разработчик 1 создаёт структуру OpenSpec в репозитории.

Он также проверяет, что команда понимает:

- где лежат active changes;
- где лежит master specification;
- где будут archived changes.

### Что делает разработчик 2

Разработчик 2 проверяет, что структура не противоречит будущей архитектуре проекта.

Например, не нужно создавать домены только по названиям технических сервисов, если поведение системы лучше описывать через бизнес-возможности.

### Что делает QA

QA проверяет, что структура specs будет удобна для поиска требований и сценариев.

### Результат шага

В проекте есть базовая структура OpenSpec.

## Шаг 3. Создать первый change

### Что происходит

Команда создаёт первый change.

Название должно быть коротким и понятным.

Хорошо:

```text
init-request-creation
init-user-authentication
init-basic-approval-flow
```

Плохо:

```text
init-system
initial-work
project-start
all-basic-stuff
```

### Что делает аналитик

Аналитик предлагает название change с точки зрения смысла задачи.

### Что делает разработчик 1

Разработчик 1 создаёт папку change и отвечает за её целостность.

Пример:

```text
openspec/changes/init-request-creation/
├── proposal.md
├── design.md
├── tasks.md
└── specs/
```

### Что делает разработчик 2

Разработчик 2 проверяет, что change не стал слишком широким.

### Что делает QA

QA проверяет, что change можно будет проверить отдельным набором сценариев.

### Результат шага

Создан первый OpenSpec change.

## Шаг 4. Заполнить proposal.md

### Что происходит

Команда описывает, зачем нужен первый slice и что именно входит в работу.

### Что делает аналитик

Аналитик является owner файла `proposal.md`.

Он описывает:

- проблему или цель;
- scope;
- out of scope;
- affected areas;
- success criteria.

### Что делает разработчик 1

Разработчик 1 проверяет proposal с точки зрения реализации.

Он добавляет вопросы:

- есть ли технические зависимости;
- есть ли неочевидные ограничения;
- нужно ли что-то вынести из scope.

### Что делает разработчик 2

Разработчик 2 проверяет proposal с точки зрения архитектурного риска.

### Что делает тестировщик 1

Тестировщик 1 проверяет, можно ли по proposal понять, что задача готова.

### Что делает тестировщик 2

Тестировщик 2 ищет неоднозначности.

Например:

```text
Что будет, если пользователь отправит пустую форму?
Что будет, если заявка создаётся повторно?
Что будет, если нет прав?
```

### Результат шага

`proposal.md` понятен всей команде.

## Шаг 5. Написать первые requirements и scenarios

### Что происходит

Команда описывает поведение системы.

Specification отвечает на вопрос:

```text
Что должна делать система?
```

Specification не отвечает на вопрос:

```text
Как это технически реализовано?
```

### Что делает аналитик

Аналитик является owner требований.

Он пишет:

- requirements;
- scenarios;
- ожидаемое поведение;
- ошибки и ограничения с точки зрения пользователя или системы.

Пример:

```md
### Requirement: Request Creation

The system SHALL allow an authorized user to create a request.

#### Scenario: Successful request creation

- GIVEN an authorized user has filled all mandatory fields
- WHEN the user creates a request
- THEN the system creates the request in Draft status
```

### Что делает тестировщик 1

Тестировщик 1 проверяет, можно ли каждый scenario протестировать.

Если scenario нельзя проверить, его нужно переписать.

### Что делает тестировщик 2

Тестировщик 2 добавляет edge cases.

Например:

```text
пустые обязательные поля
слишком длинные значения
повторная отправка формы
пользователь без прав
```

### Что делает разработчик 1

Разработчик 1 проверяет, что требования технически реализуемы.

### Что делает разработчик 2

Разработчик 2 проверяет, нет ли требований, которые ломают архитектурные ограничения или создают опасный precedent.

### Результат шага

Есть первый `spec.md` с requirements и scenarios.

## Шаг 6. Написать design.md

### Что происходит

Разработчики описывают технический подход.

Design отвечает на вопрос:

```text
Как мы собираемся это реализовать?
```

На этом же шаге команда предварительно отмечает решения, которые могут стать ADR.

ADR — это Architecture Decision Record, то есть короткий документ с важным архитектурным решением: какой вариант выбрали, почему выбрали именно его и какие последствия это создаёт для будущей разработки.

### Что делает разработчик 1

Разработчик 1 является owner файла `design.md`.

Он описывает:

- текущее поведение, если оно есть;
- proposed solution;
- affected components;
- API impact;
- data impact;
- security impact;
- риски;
- технические ограничения;
- потенциальные ADR.

Отдельно в `design.md` нужно добавить раздел:

```md
## Potential ADR

- Нужно ли сохранить это решение как постоянное архитектурное правило?
- Повлияет ли это решение на будущие задачи?
- Нужно ли ссылаться на это решение из master spec?
```

### Что делает разработчик 2

Разработчик 2 проводит architecture review.

Он проверяет:

- не противоречит ли решение архитектуре;
- нет ли лишней сложности;
- нет ли опасного временного решения;
- нужно ли сразу создать ADR;
- нужно ли будет добавить ссылку на ADR в master spec;
- не нарушены ли enterprise-ограничения.

### Что делает аналитик

Аналитик читает design не ради технических деталей, а чтобы проверить:

- не потерян ли бизнес-смысл;
- не изменился ли scope;
- не появилось ли ограничение, важное для заказчика;
- не нужно ли отразить часть решения как наблюдаемое поведение в master spec.

### Что делает QA

QA читает design, чтобы понять:

- какие компоненты затронуты;
- где нужны ручные проверки;
- где нужен regression;
- какие технические риски нужно проверить;
- нужно ли сохранить новые regression rules рядом с ADR или testing docs.

### Результат шага

Есть согласованный `design.md`.

Также команда понимает, какие решения потенциально станут ADR.

## Шаг 7. Разбить работу в tasks.md

### Что происходит

Команда превращает proposal, spec и design в список конкретных задач.

### Что делает разработчик 1

Разработчик 1 является owner файла `tasks.md`.

Он разбивает работу на блоки:

```text
Specification
Design
Implementation
Testing
Review
Archive
```

### Что делает разработчик 2

Разработчик 2 проверяет, что задачи не скрывают архитектурно важные изменения.

### Что делает аналитик

Аналитик проверяет, что все важные требования представлены в задачах.

### Что делает тестировщик 1

Тестировщик 1 добавляет QA tasks:

```text
подготовить тестовые данные
проверить основной сценарий
проверить ошибки
проверить regression
```

### Что делает тестировщик 2

Тестировщик 2 добавляет exploratory / edge-case проверки.

### Результат шага

Есть `tasks.md`, по которому можно работать.

## Шаг 8. Провести review до реализации

### Что происходит

Перед началом кода команда делает короткий review.

Цель — убедиться, что все одинаково понимают задачу.

### Что проверяет аналитик

```text
Смысл задачи понятен.
Scope не расползся.
Out of scope явно указан.
```

### Что проверяет разработчик 1

```text
Реализация понятна.
Tasks достаточно конкретны.
Нет неизвестных технических блокеров.
```

### Что проверяет разработчик 2

```text
Архитектурный подход приемлем.
Ограничения учтены.
Нет опасных shortcuts.
```

### Что проверяет тестировщик 1

```text
Scenarios проверяемы.
Manual test plan понятен.
Regression scope понятен.
```

### Что проверяет тестировщик 2

```text
Edge cases не забыты.
Есть негативные сценарии.
Есть проверки нестандартного поведения.
```

### Результат шага

Команда даёт go/no-go.

Если есть серьёзные вопросы, код не начинается.

## Шаг 9. Реализовать первый slice

### Что происходит

Разработчик 1 реализует изменение по `tasks.md`.

### Что делает разработчик 1

Он реализует код и поддерживает OpenSpec-файлы актуальными.

Если во время реализации меняется подход, он обновляет `design.md`.

Если меняется поведение, он обновляет `spec.md`.

### Что делает разработчик 2

Он может подключаться к сложным вопросам до PR, чтобы не переносить архитектурный спор в самый конец.

### Что делает аналитик

Аналитик отвечает на вопросы по смыслу и scope.

### Что делает QA

QA готовит проверки и тестовые данные.

### Результат шага

Код готов к PR.

## Шаг 10. Провести PR Review

### Что происходит

PR проверяется не только как код.

Проверяется вся цепочка:

```text
proposal → spec → design → tasks → code → tests
```

### Что делает разработчик 1

Разработчик 1 создаёт PR и описывает:

- ссылку на OpenSpec change;
- какие requirements реализованы;
- какие design decisions важны;
- что протестировано;
- какие риски остались.

### Что делает разработчик 2

Разработчик 2 является основным reviewer.

Он проверяет:

- качество кода;
- соответствие design;
- архитектурные ограничения;
- отсутствие скрытого scope;
- необходимость ADR.

### Что делает аналитик

Аналитик смотрит PR, если есть смысловые изменения.

### Что делает QA

QA проверяет, что PR содержит достаточно информации для тестирования.

### Результат шага

PR получил техническое approval или вернулся на доработку.

## Шаг 11. Провести QA

### Что происходит

QA проверяет реализованное поведение.

### Что делает тестировщик 1

Тестировщик 1 выполняет основной test plan:

- happy path;
- negative cases;
- regression;
- проверка ошибок;
- проверка прав, если применимо.

### Что делает тестировщик 2

Тестировщик 2 выполняет независимую проверку:

- edge cases;
- exploratory testing;
- нестандартные последовательности действий;
- попытки сломать сценарий.

### Что делает разработчик 1

Разработчик 1 исправляет найденные дефекты.

### Что делает разработчик 2

Разработчик 2 проверяет сложные исправления, если они затрагивают архитектуру.

### Что делает аналитик

Аналитик помогает решить спорные вопросы ожидаемого поведения.

### Результат шага

QA даёт sign-off или возвращает задачу на доработку.

## Шаг 12. Выполнить Design Harvest

### Что происходит

Перед archive команда смотрит, какие знания должны жить дальше.

Design Harvest нужен, чтобы важные решения не остались только в `design.md` или PR-комментариях.

### Что делает разработчик 1

Разработчик 1 проверяет `design.md` и предлагает, что нужно вынести в постоянные документы:

- ADR;
- architecture docs;
- development conventions;
- testing docs;
- master spec.

### Что делает разработчик 2

Разработчик 2 решает, нужен ли ADR.

ADR нужен, если решение:

- влияет на будущую разработку;
- задаёт архитектурное ограничение;
- объясняет, почему выбран именно такой подход;
- будет важно при следующих PR-review;
- может вызвать вопросы через несколько месяцев.

Пример ADR:

```text
ADR-001: Request status transitions must use transition service
```

Пример решения:

```text
Все изменения статуса должны проходить через StatusTransitionService.
```

### Что делает аналитик

Аналитик проверяет, что master spec отражает финальное поведение.

Если техническое решение влияет на наблюдаемое поведение, оно должно попасть в master spec как требование.

Если решение не является поведением, но важно для понимания ограничения, в master spec можно добавить ссылку на ADR.

### Что делает QA

QA проверяет, нужно ли сохранить новые regression checks.

Если ADR создаёт постоянное ограничение, QA проверяет, появились ли постоянные проверки, которые нужно учитывать в будущих задачах.

### Результат шага

Команда решила:

- что попадёт в master spec;
- какой ADR нужно создать;
- нужна ли ссылка из master spec на ADR;
- что останется только в archived change;
- что нужно добавить в testing docs.

## Шаг 13. Выполнить archive

### Что происходит

Change архивируется.

После archive актуальное поведение должно быть в `openspec/specs/`.

### Что делает разработчик 1

Разработчик 1 выполняет archive и проверяет результат.

### Что делает разработчик 2

Разработчик 2 проверяет, что важные design decisions не потерялись.

### Что делает аналитик

Аналитик проверяет master spec.

### Что делает QA

QA подтверждает, что scenarios соответствуют проверенному поведению.

### Результат шага

Первый OpenSpec cycle завершён.

## Definition of Done для первого change

Первый change считается завершённым, если:

```text
[ ] proposal.md написан и согласован
[ ] spec.md содержит requirements и scenarios
[ ] design.md описывает технический подход
[ ] design.md содержит Potential ADR, если есть важные решения
[ ] tasks.md закрыт
[ ] код реализован
[ ] PR прошёл review
[ ] QA дал sign-off
[ ] Design Harvest выполнен
[ ] ADR создан, если появилось постоянное архитектурное решение
[ ] master spec обновлён
[ ] master spec содержит ссылку на ADR, если ADR важен для понимания поведения или ограничения
[ ] archive выполнен
```

## Хороший первый change

```text
init-user-authentication
init-request-creation
init-basic-approval-flow
```

## Плохой первый change

```text
init-whole-system
create-all-backend
full-platform-implementation
```

## Главное правило для нового проекта

Не пытайтесь сделать идеальную спецификацию заранее.

Лучше пройти один маленький полный цикл и научиться работать правильно.
